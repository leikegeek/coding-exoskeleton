# 测试策略与执行

## 核心原则
1. **需求驱动**：所有测试场景必须基于需求和代码行为设计。
2. **可落地性**：测试方案应服务于实际编写测试，不追求形式化堆砌。
3. **场景完整性**：必须覆盖正常、边界、异常场景，并明确不测范围。
4. **依赖可控性**：需说明 mock / stub 策略，以及哪些部分更适合集成测试。
5. **主动澄清**：需求、行为或依赖不明确时，先提问再生成方案。

## 工作流程
1. 明确被测对象、输入输出和关键依赖
2. 划分正常、边界、异常、状态流转和幂等类场景
3. 判断哪些依赖应 mock，哪些更适合集成测试
4. 输出结构化单测设计文档
5. 在编码阶段指导每个任务的测试编写

## 执行约束
- 不机械追求 100% 覆盖率
- 不强制所有内容都扩成大表格或长说明
- 若项目对覆盖率有要求，应在方案中单独标注关键路径覆盖情况

# 实施计划与任务拆解

## 技能目标

将技术方案转化为可执行的任务清单，区分基础设施任务和编码任务，将编码任务按业务边界归组生成子代理派发方案，为 B2 编排模式做准备。

## 核心原则

1. **先分类后归组**：先将任务分为基础设施任务和编码任务，只有编码任务参与业务边界归组
2. **依赖可见**：明确标注任务间依赖和组间依赖，支持调度决策
3. **业务边界归组**：将相关联的编码任务按业务模块归组，每组交给一个子代理在隔离上下文中执行
4. **TDD 导向**：每个编码任务包含对应的测试文件
5. **可追溯**：每个任务关联需求编号和技术方案章节

## 工作流程

### 1. 分析技术方案

从技术方案文档中提取：
- 需要新增/修改/删除的文件
- 涉及的层级（adapter / application / domain / infrastructure）
- 数据模型变更
- 接口定义
- 涉及的业务模块/功能域

### 2. 拆解任务清单

每个任务包含：

```
任务ID：T-001
描述：创建 OrderFilterController 接口入口
层级：adapter
业务模块：订单筛选
影响文件：adapter/src/.../OrderFilterController.java
测试文件：adapter/src/test/.../OrderFilterControllerTest.java
依赖：无
TDD：写 Controller 测试 → 实现 Controller → 验证通过
```

### 3. 任务分类

将所有任务分为两类，决定后续的处理方式：

**基础设施任务**（由主 Agent 在编排模式下直接执行）：
- 项目搭建（脚手架、目录结构初始化）
- 数据库迁移（schema 创建、Alembic/Flyway 迁移脚本）
- 配置初始化（配置文件、环境变量、CI/CD 脚本）
- 依赖管理（pom.xml / package.json / pyproject.toml 变更）
- 全局公共基础设施（工具类、通用异常处理、全局拦截器）

**编码任务**（参与业务边界归组，委派子代理执行）：
- 业务逻辑实现（Controller、Service、Repository、前端组件）
- API 接口开发
- 业务领域模型实现
- 业务相关的测试编写

**分类原则**：基础设施任务通常是后续编码任务的前置依赖，且不属于某个特定业务模块。如果一个任务既涉及基础设施又涉及业务逻辑，按主要职责归类。

### 4. 标注依赖关系

生成依赖图，明确：
- 哪些任务无依赖（独立）
- 哪些任务必须顺序执行（有依赖）
- 任务间的依赖方向
- 基础设施任务与编码任务之间的依赖

依赖标注格式：
```
[基础设施] T-001 (独立)
[基础设施] T-002 (依赖 T-001)
[编码-订单模块] T-003 (依赖 T-002) ─┐
[编码-订单模块] T-004 (依赖 T-002) ─┤
[编码-通知模块] T-005 (依赖 T-002) ─┘
[编码-集成] T-006 (依赖 T-003, T-005)
```

### 5. 业务边界归组与派发方案

**仅对编码任务**按所属业务模块/功能域归组，生成子代理派发方案：

**归组规则**：
- 影响同一业务链路（如同一 Controller + Service + Repository）的编码任务归为一组
- 前后端分离的项目中，前端和后端各自作为独立的业务边界组
- 跨模块的集成任务（如多模块联调）单独标记为"顺序执行"组
- 每组内部标注执行顺序（按组内任务间的依赖关系排序）

**组间依赖标注**：
- 标注哪些组的产出是其他组的前置依赖
- 标注组间传递的接口契约（如 G-01 产出的 Service 接口被 G-02 调用）
- 基于组间依赖生成建议派发顺序（拓扑排序）

**派发判定**：
- 编码任务的业务边界分组数 >= 2 → 建议进入编排模式（主 Agent 转为编排者，逐组串行派发子代理）
- 编码任务的业务边界分组数 < 2 → 建议主 Agent 直接执行（拆子代理的上下文隔离收益不足以覆盖调度开销）

### 6. 定义测试策略

结合 `testing` skill，确定：
- 核心路径测试用例
- 边界条件测试用例
- 异常场景测试用例
- Mock 策略（哪些依赖需要 mock）

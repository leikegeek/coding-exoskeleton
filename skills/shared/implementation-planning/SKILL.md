# 实施计划与任务拆解

## 技能目标

将技术方案转化为可执行的任务清单，每个任务粒度适中（2-5 分钟），并标注依赖关系以支持 Subagent 并行执行。

## 核心原则

1. **细粒度**：每个任务聚焦一个明确的改动点，2-5 分钟可完成
2. **依赖可见**：明确标注任务间依赖，支持并行调度
3. **TDD 导向**：每个任务包含对应的测试文件
4. **可追溯**：每个任务关联需求编号和技术方案章节

## 工作流程

### 1. 分析技术方案

从技术方案文档中提取：
- 需要新增/修改/删除的文件
- 涉及的层级（adapter / application / domain / infrastructure）
- 数据模型变更
- 接口定义

### 2. 拆解任务清单

每个任务包含：

```
任务ID：T-001
描述：创建 OrderFilterController 接口入口
层级：adapter
影响文件：adapter/src/.../OrderFilterController.java
测试文件：adapter/src/test/.../OrderFilterControllerTest.java
依赖：无（独立任务，可并行）
TDD：写 Controller 测试 → 实现 Controller → 验证通过
```

### 3. 标注依赖关系

生成依赖图，明确：
- 哪些任务可以并行（无依赖）
- 哪些任务必须顺序执行（有依赖）
- 建议的执行顺序

依赖标注格式：
```
T-001 (独立) ─┐
T-002 (独立) ─┼─ 并行组 A
T-003 (独立) ─┘
T-004 (依赖 T-001) ─┐
T-005 (依赖 T-002) ─┼─ 并行组 B（等组 A 完成）
T-006 (依赖 T-004, T-005) ── 顺序执行
```

### 4. 定义测试策略

结合 `testing` skill，确定：
- 核心路径测试用例
- 边界条件测试用例
- 异常场景测试用例
- Mock 策略（哪些依赖需要 mock）

# /code — 流水线 B 入口：从技术方案开始编码

当用户输入 `/code` 时，启动流水线 B（技术方案 → 编码 → 测试 → 交付）。

## 触发方式

- `/code SV-34577` — 指定需求编号，AI 查找对应技术方案文档
- `/code SV-34577 @docs/design/SV-34577-tech-design.md` — 指定需求编号和技术方案文件
- `/code` + 粘贴技术方案内容 — 直接提供文档内容

## 预检步骤（自动执行，对用户透明）

> **关于 Hooks**：Hooks 治理环境由 `install.ps1` 在安装插件时统一写入 `~/.cursor/hooks.json`（全局用户级），无需每个项目单独安装。

### 预检：项目画像检查

检查当前项目是否已生成项目画像文件：

1. 检查项目根目录下是否存在 `AGENTS.md` 文件
2. **如果已存在**：读取 `AGENTS.md`，将其中的技术栈、架构模式、模块结构等信息作为项目上下文，进入正式流程
3. **如果不存在**：暂停当前流程，引导用户完成项目初始化
   - 告知用户：「检测到当前项目尚未初始化项目画像，需要先完成初始化以确保 AI 准确理解项目上下文。正在启动 /init 流程...」
   - 执行 `/init` 命令的完整流程（读取 `init.md`）
   - 完成后告知用户：「项目画像已就绪，继续执行 /code 流程...」
   - 读取刚生成的 `AGENTS.md`，然后继续正式流程

## 两种进入方式

### 方式一：从流水线 A 衔接

用户在 `/start` 流程中选择了"继续编码"。此时上下文已就绪，跳过理解和确认环节，直接进入实施准备。

### 方式二：独立进入（关键流程）

拿到已过审的技术方案文档（自己之前写的，或架构师/其他开发者输出的），直接编码。

**必须执行：先理解，再确认，后编码。**

#### B0: 技术方案接入

1. **要求需求编号**：从 `AGENTS.md` 的 `需求编号正则` 字段读取匹配规则（默认 `SV-\d+`），检查是否提供了需求编号。未提供则要求补充：

> 请提供需求编号（如 SV-34577，或你们团队约定的编号格式），它是串联技术方案、分支、变更清单的唯一 key。

2. **理解技术方案文档**：完整阅读并提炼：
   - 需求背景和目标
   - 核心技术方案（架构、数据模型、接口设计）
   - 影响范围和约束

3. **识别模糊/不理解之处**：主动列出所有疑问，分三类：
   - **歧义**：文档中存在多种理解方式的描述
   - **缺失**：文档未覆盖但编码必需的细节（异常处理策略、边界值处理等）
   - **矛盾**：文档内部或与项目现状不一致的地方

4. **与用户逐一确认**：展示疑问列表，全部确认后才继续

5. **检查实施进度**：检查技术方案文档中是否存在「八、实施进度」段落
   - **如果存在**：这是一个进行中的需求。读取进度段落，了解当前状态和任务清单，跳过已完成的任务，从未完成处继续
   - **如果不存在**：这是首次编码，按正常流程执行

6. 读取 `AGENTS.md` 获取项目上下文（技术栈、架构模式、模块结构），建立任务契约（编码模式）

## 执行流程

> **阶段切换压缩提示**：B0→B1、B1→B2、B2→B3、B3→B4 每个阶段切换点，读取 `context-compaction` skill，评估是否需要执行上下文压缩。

### B1: 实施准备

1. 切换到**编码模式**
2. 创建 feature 分支：`feature/SV-xxxxx-brief-description`
3. 拆解任务清单（读取 `implementation-planning` skill）：
   - 明确影响文件、所属层级、业务模块、测试文件
   - 标注任务间依赖关系
   - **任务分类**：区分基础设施任务和编码任务
   - **对编码任务按业务边界归组，生成子代理派发方案**（包含分组、组内顺序、组间依赖、建议派发顺序）
4. 定义测试策略（读取 `testing` skill）
5. 初始化变更记录文档 `docs/delivery/SV-xxxxx-changelist.md`（如目录不存在则自动创建；读取 `coding` skill 中的文档骨架）
6. 初始化实施进度段落：在技术方案文档（`docs/design/SV-xxxxx-tech-design.md`）末尾创建「八、实施进度」段落，填入任务清单，状态设为「编码中」

### B2: 编码实施

#### B2.0 派发判定

读取 B1 产出的子代理派发方案，判定执行模式：

- **编码任务的业务边界分组数 >= 2** → 进入**编排模式**：主 Agent 转为编排者角色，不直接编写业务代码，先执行基础设施任务，再逐组派发编码子代理
- **编码任务的业务边界分组数 < 2** → 进入**直接执行模式**：主 Agent 按现有 TDD + Record 节奏自行完成所有任务（流程同下方"直接执行模式"）

进入编排模式前，向用户展示派发方案并确认。**默认：组间串行**（完成并验收 G-01 后，再派发 G-02，依此类推）。**禁止**在用户未**明确、单独**确认前，同时启动两个及以上编码子代理的 Task 调用。

向用户沟通时可采用如下结构（先串行、并行仅作可选项，避免被误解为「默认并行再确认是否串行」）：

> **执行方式（默认）**：**组间串行** —— 基础设施任务完成后，按 G-01 → G-02 → … 依次派发；**每一时刻最多只有一组**编码子代理在跑。
> **并行（可选，需显式授权）**：仅当你明确回复 **「允许并行」** 时，才可对 **组间无依赖** 的多个组**同时**派发；否则一律按串行继续。
>
> 本次共 N 个任务，其中基础设施任务 X 个（主 Agent 直接执行），编码任务按业务边界分为 M 组：
> - G-01: {业务边界} — {任务列表}
> - G-02: {业务边界} — {任务列表}
> - ...
> 建议派发顺序：基础设施任务 → G-01 → G-02 → …
>
> 请确认：是否采纳上述 **默认串行** 的派发与顺序，或说明需要 **允许并行** 的组与理由。

#### B2.1 编排模式：基础设施任务先行

主 Agent 先直接执行所有基础设施任务（项目搭建、数据库迁移、配置初始化等），为后续编码子代理建立可工作的代码基座。

基础设施任务遵循与直接执行模式相同的 TDD + Record 节奏。全部完成后，commit 并更新变更记录和实施进度，然后进入派发循环。

#### B2.2 编排模式：派发循环

主 Agent 按确认的派发顺序，**逐组串行**执行以下循环：

**第一步：准备上下文**
- 按 `coding-subagent` agent 的上下文边界，为当前组准备最小信息集：
  - 本组任务清单（含组内执行顺序）
  - 技术方案文档中与本组直接相关的章节（摘取，非全文）
  - `AGENTS.md` 项目画像
  - 需求编号和分支名
  - 前序组的接口契约（如果本组依赖前序组产出）
- 剔除禁止传递的内容：其他组的任务清单、设计讨论历史、编排决策过程

**第二步：派发子代理**
- 使用 Task 工具启动 `generalPurpose` 子代理
- **每次循环只派发一组**：对本循环目标组**仅启动一个** Task；**不要**在串行方案下、或在用户未明确回答「**允许并行**」前，**同时**为 G-01、G-02… 各启动一个 Task（那是并行，不是本流程默认行为）。
- prompt 中包含：任务清单、技术方案相关章节、AGENTS.md 内容、TDD 节奏要求、`coding-subagent` agent 的返回格式要求
- 子代理在隔离的上下文中按 TDD 节奏逐任务执行编码

**第三步：验收**
- 子代理完成后，主 Agent 收集返回的结构化结果
- 验收检查项：
  - 每个任务的完成状态是否全部成功
  - 子代理自行做出的假设决策是否合理（不合理的标记为需修复）
  - 测试是否全部通过
  - 变更是否在任务范围内（无超范围改动）

**第四步：记录与推进**
- 验收通过：将本组变更记录追加到 `docs/delivery/SV-xxxxx-changelist.md`，更新实施进度
- 验收不通过：决策"重新派发同组（附修复指示）"或"退出编排模式，主 Agent 接管剩余任务"
- 退出编排模式必须向用户说明原因

**第五步：下一组**
- 还有未派发的组 → 回到第一步，派发下一组
- 所有组完成 → 进入收敛

> **并行选项**：如果用户在确认派发方案时显式选择了"允许并行"，可同时派发无组间依赖的多组。但默认始终串行。

#### B2.3 编排模式：收敛

所有组完成并验收后：
1. 确认所有变更记录已合并到 `docs/delivery/SV-xxxxx-changelist.md`
2. 更新实施进度为全部编码完成
3. 运行全量集成测试，确认各组代码协同工作
4. 执行逻辑层面的冲突检查（如有并行执行的组）
5. 向用户报告 B2 完成摘要（各组完成状态、总变更文件数、测试通过率）

#### B2.4 直接执行模式（编码任务分组数 < 2）

主 Agent 自行按 TDD + Record 节奏完成编码（读取 `coding` skill；每个任务 commit 后，委派 `tdd-guide` agent 检查 TDD 纪律）：
1. 写单元测试（红）
2. 运行测试，确认失败
3. 写最小实现代码（绿）
4. 运行测试，确认通过
5. 提交（commit message 关联 SV-ID）
6. **更新变更记录文档**：将本次 commit 的变更追加到 `docs/delivery/SV-xxxxx-changelist.md`，高风险变更即时标注

### B3: 代码审查与对齐

读取 `verification-loop` skill，以结构化验证循环执行以下检查。

#### 3.1 全量测试

运行全量单元测试。**门禁**：测试不通过，不允许继续。

#### 3.2 性能检查

读取 `performance-analysis` skill，执行性能扫描。

#### 3.3 三方对齐审查

以三份材料为输入，执行系统化的对齐检查（读取 `code-reviewer` agent）：

| 输入材料 | 来源 | 用途 |
|----------|------|------|
| 变更记录文档 | 编码过程中增量维护的 `docs/delivery/SV-xxxxx-changelist.md` | 开发者视角的变更说明 |
| git diff | `git diff main...HEAD` 的实际代码变更 | 代码事实 |
| 技术方案文档 | `docs/design/SV-xxxxx-tech-design.md` | 业务意图 |

检查维度：
- **变更记录 vs git diff**：所有实际变更是否都被记录？有无遗漏或多余的文件？
- **变更记录 + diff vs 技术方案**：代码是否在业务需求上跑偏？有无超出方案范围的变更？有无方案要求但未实现的功能？
- **架构规范、代码质量、测试覆盖、高风险变更**（原有评审维度保留）

#### 3.4 产出

审查阶段产出三份文档：

1. **代码评审报告**：问题列表（Critical / Warning / Info）+ 评审结论（保存到 `docs/delivery/SV-xxxxx-review-report.md`）
2. **变更清单（定稿）**：对编码阶段积累的变更记录做最终格式化，补齐遗漏，确认与 diff 一致
3. **技术参考文档**：从变更记录和审查结果中提炼，面向测试人员（保存到 `docs/delivery/SV-xxxxx-tech-ref.md`）

**门禁**：发现 Critical 级别的业务偏差时，必须修复后重新审查。

验证循环全部 PASS 后，更新实施进度：勾选 B3 任务项，状态设为「已验证」，更新最后更新时间。

### B4: 交付

审查通过后，委派 `doc-updater` agent 检查交付文档的完整性、格式合规性和与代码变更的一致性。检查通过后，更新实施进度：勾选 B4 任务项，状态设为「已交付」，更新最后更新时间。然后展示所有交付物的摘要：

> ## 交付物已就绪
>
> - 变更清单：`docs/delivery/SV-xxxxx-changelist.md`
> - 技术参考文档：`docs/delivery/SV-xxxxx-tech-ref.md`
> - 代码评审报告：`docs/delivery/SV-xxxxx-review-report.md`
> - 代码：feature/SV-xxxxx-... 分支
>
> 请选择：
> 1. **提交 PR** — 自动填写 PR 描述，附变更清单和技术参考文档
> 2. **本地合并** — 合并到当前主分支
> 3. **保留分支** — 不合并，保留分支供后续处理

## 关键约束

- 编码模式下，Hooks 允许构建/测试命令，但拦截危险操作
- 所有 commit message 必须关联需求编号
- 编码子代理同样受 Rules 约束（架构分层、命名规范、事务管理、性能约束等）
- 变更记录文档在编码过程中持续更新，不在交付阶段事后生成
- **编排模式约束**：进入编排模式后，主 Agent 禁止直接编写业务代码，职责限定为拆分、派发、验收、收敛。如需接管，必须显式退出编排模式并向用户说明原因

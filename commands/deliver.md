# /deliver — 交付物补救生成

当用户输入 `/deliver` 时，检查并补救交付物。

> **正常流程无需使用此命令。** 在 `/code` 流程中，变更清单在编码阶段增量维护，技术参考文档在代码审查阶段自动产出，交付环节（B4）直接展示已就绪的交付物。

## 适用场景

- 交付文档意外丢失或损坏，需要重新生成
- 未经 `/code` 流程的编码工作，需要补充生成交付文档
- 对已有交付文档进行格式修正或内容补充

## 前置条件

- 当前在 feature 分支上
- 有相对于基准分支的代码变更
- 项目已完成初始化（`AGENTS.md` 存在）。如果未初始化，先引导用户执行 `/init`
- Hooks 治理环境已安装（`~/.cursor/hooks.json` 存在，由插件 `install.ps1` 全局写入）。如果未安装，提示用户在插件目录执行 `install.ps1`

## 执行流程

### 第一步：检测已有交付物

检查 `docs/delivery/` 目录下是否已存在当前需求的交付文档：

1. **如果变更清单和技术参考文档都存在**：读取 `delivery` skill，执行格式化校验和 diff 对账，补齐遗漏
2. **如果部分存在**：保留已有文档，仅补充生成缺失的部分
3. **如果都不存在**：进入完整的补救生成模式（见下方）

### 第二步：补救生成（仅当文档不存在时）

> 注意：补救模式下 AI 需要从 diff 回溯理解变更意图，文档精度可能低于正常流程中的增量记录。

读取 `AGENTS.md` 获取项目上下文（技术栈、架构模式、模块结构），辅助理解变更意图。

**生成变更清单**：

从 `git diff main...HEAD`（或对比基准分支）提取所有变更：

1. 按类型分组列出所有变更文件（新增/修改/删除）
2. 读取每个变更文件，推断功能说明
3. 扫描高风险变更（方法移除、代码注释、接口签名变更）
4. 保存到 `docs/delivery/SV-xxxxx-changelist.md`

**生成技术参考文档**：

读取变更文件内容，生成面向测试人员的技术参考文档：

1. 接口清单（URL、方法、请求/响应结构）
2. 核心功能说明
3. 数据流概要
4. 配置变更
5. 注意事项

保存到 `docs/delivery/SV-xxxxx-tech-ref.md`

### 第三步：展示交付物并让用户选择

展示所有交付物的摘要，然后询问：

> ## 交付物已就绪
>
> - 变更清单：`docs/delivery/SV-xxxxx-changelist.md`
> - 技术参考文档：`docs/delivery/SV-xxxxx-tech-ref.md`
> - 代码评审报告：`docs/delivery/SV-xxxxx-review-report.md`（如存在）
> - 代码：feature/SV-xxxxx-... 分支
>
> 请选择：
> 1. **提交 PR** — 自动填写 PR 描述，附变更清单和技术参考文档
> 2. **本地合并** — 合并到当前主分支
> 3. **保留分支** — 不合并，保留分支供后续处理

## 关键约束

- 变更清单必须覆盖所有 git diff 中的文件
- 被移除和注释掉的方法必须在"需关注"中单独列出
- 技术参考文档面向测试人员，语言应清晰易懂
- PR 描述自动关联需求编号和交付文档
- 补救生成时应提示用户：此文档为事后回溯生成，建议人工复核

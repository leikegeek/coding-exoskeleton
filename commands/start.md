# /start — 流水线 A 入口：从需求开始

当用户输入 `/start` 时，启动流水线 A（需求 → 技术方案）。

## 触发方式

- `/start SV-34577` — 带需求编号
- `/start SV-34577 需求文档内容...` — 带需求编号和文档
- `/start 给订单列表加一个按供应商筛选的功能` — 口头需求（将要求补充 SV-ID）

## 执行流程

### 预检步骤（自动执行，对用户透明）

检查当前项目是否已生成项目画像文件：

1. 检查项目根目录下是否存在 `AGENTS.md` 文件
2. **如果已存在**：读取 `AGENTS.md`，将其中的技术栈、架构模式、模块结构等信息作为项目上下文，进入第一步
3. **如果不存在**：暂停当前流程，引导用户完成项目初始化
   - 告知用户：「检测到当前项目尚未初始化项目画像，需要先完成初始化以确保 AI 准确理解项目上下文。正在启动 /init 流程...」
   - 执行 `/init` 命令的完整流程（读取 `init.md`）
   - 完成后告知用户：「项目画像已就绪，继续执行 /start 流程...」
   - 读取刚生成的 `AGENTS.md`，然后继续第一步

> **关于 Hooks**：Hooks 治理环境由 `install.ps1` 在安装插件时统一写入 `~/.cursor/hooks.json`（全局用户级），无需每个项目单独安装。

### 第一步：需求录入

1. 从 `AGENTS.md` 的 `需求编号正则` 字段读取匹配规则（默认 `SV-\d+`），检查用户输入中是否包含需求编号
2. **有需求编号**：解析需求文档，提取编号、标题、描述、验收标准
3. **无需求编号**：提炼需求要素，然后要求用户提供需求编号。需求编号是跨团队协作的 key，不可省略。提示用户：

> 口头需求已收到。为了保证全链路可追溯，请提供需求编号（如 SV-34577，或你们团队约定的编号格式）。如果还没有正式编号，请提供一个团队内可识别的标识。

4. 读取项目上下文：优先使用 `AGENTS.md` 中的项目画像（技术栈、架构模式、模块结构），仅在画像信息不足时补充扫描构建文件
5. 建立任务契约，进入**设计模式**：
   - 模式：设计模式
   - 需求编号：必填
   - 允许写入路径：仅文档目录
   - 禁止项：不修改项目代码、不执行构建命令

### 第二步：技术方案设计

读取并执行 `tech-design` skill：

1. 逐个澄清问题（一次一个，优先选择题）
2. 识别受影响的模块和层级
3. 提出 2-3 种方案，附取舍分析和推荐
4. 输出结构化技术方案文档
5. 执行方案自评审（读取 `design-review` skill）
6. 若方案涉及新增模块、引入新技术组件、跨模块数据流变更或数据模型重构，委派 `architect` agent 执行架构决策审查
7. 等待用户确认方案

### 第三步：出口选择

方案经用户确认后，读取 `context-compaction` skill，评估是否需要执行上下文压缩（尤其是选择「继续编码」时，A→B 衔接是关键压缩点）。

然后询问：

> 技术方案已确认。请选择：
> 1. **继续编码** — 直接进入编码阶段（流水线 B）
> 2. **断开** — 仅保存技术方案文档，后续通过 `/code SV-ID` 继续

- 选择"继续编码"：切换到编码模式，自动进入流水线 B
- 选择"断开"：保存文档到 `docs/design/SV-xxxxx-tech-design.md`（如目录不存在则自动创建），结束当前流程

## 关键约束

- 设计模式下，Hooks 会拦截构建/提交类命令
- 技术方案文档中必须包含需求编号
- 断开后，技术方案文档是续接流水线 B 的唯一依据
